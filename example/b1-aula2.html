Componentes Básicos e Tela de Login

Entendendo a arquitetura de Views textuais, inputs interativos e formulários baseados em estado para construir a principal tela de entrada de dados de um app corporativo.

⚠️ Regra de Ambiente: Emulador

Lembre-se: todo código desenvolvido nesta disciplina deve ser executado no **Android Virtual Device (AVD)**. A visualização do teclado nativo cobrindo os botões (teclado virtual do emulador) será essencial para o debug desta aula.

🎯 Objetivos de Aprendizagem

  • Dominar os Componentes Primitivos: Compreender com profundidade o papel do `View`, `Text`, `TextInput` e `TouchableOpacity`.
  • Gerenciar Estados Locais: Utilizar a desestruturação do React Hook `useState` para capturar os dados assincronamente da tela (E-mail e Senha).
  • Tratar o Teclado Nativo: Utilizar `KeyboardAvoidingView` para garantir que o teclado digital da tela não esconda o botão de "Entrar" durante o preenchimento.
  • Validar Dados do Usuário: Aplicar lógica de sanitização básica impedindo logins vazios ou senhas excessivamente curtas e prover alertas nativos.
  • Construir um Layout Realista: Agrupar componentes numa hierarquia complexa que simule o formulário rígido do acesso padrão corporativo.

📖 Teoria: As Peças Geométricas do React Native

Quando abandonamos o ambiente da Web, deixamos de lado as familiares tags HTML (`<div>`, `<span>`, `<button>`). O React Native provê o que chamamos de Componentes Core (Essenciais). Esses componentes não desenham sozinhos a tela, mas eles sinalizam via The Bridge / Fabric qual peça de UI nativa o Android ou iOS devem invocar da sua memória. A base fundacional dessa construção é a View.

A <View> é o análogo direto à div do HTML, mas conceitualmente mais robusta. No Android, uma `View` se traduz para um android.view.ViewGroup, que é a superclasse base de todo o desenho gráfico do pacote do Google. Ela serve para envolver, estilizar, empilhar hierarquicamente e aplicar lógica de acessibilidade a agrupamentos. Sozinha, ela é invisível num sistema de coordenadas (a menos que apliquemos cor/fundo e altura a ela).

Ao contrário da web, no mobile nós somos proibidos de largar um texto bruto solto dentro de um container. Se você inserir "Olá SENAI" dento de uma <View>, nosso motor Metro emitirá um Red Screen of Death (Erro fatal) declarando que os caracteres devem interagir somente mediante à sua estrutura correspondente: o componente Text. O componente Text mapeia uma string tipográfica para a classe nativa android.widget.TextView.

No desenrolar da criação de uma tela corporativa de Login — requisito zero de qualquer sistema ERP interno — nós necessitamos interagir com a digitação do dedo sujo de graxa do mecânico ou a digitação rápida no pátio de obras. Para o recebimento de teclado, entra o vital <TextInput>. Esse componente não só mapeia para o android.widget.EditText, mas herda propriedades fundamentais de comportamento: qual layout o teclado virtual deve exibir (Mudar de QWERTY genérico para Teclado Numérico ou Teclado de E-mail com arroba fácil). Essa especificidade reduz a fricção e os erros de preenchimento de campo durante o turno operacional.

Diferentemente da web que possuí formulários dinâmicos de submissão (tags de <form>), no ecossistema mobile não enviamos a tela inteira. Nós escutamos cada tecla batida em propriedades de evento como onChangeText e salvamos essa tecla em tempo real (estado local ou State). Esse processo é conhecido por formulário controlado, onde a UI (texto digitado visível na caixa) é uma pintura direta da variável gerada pelo componente abstrato (o estado useState carregado na memória RAM do React).

E quando falamos de botões? O componente <Button> padrão do pacote Core é problemático! O Facebook/Meta (criador do React Native) o desenhou para ser rápido: ele delega o design inteiramente para o Material UI nativo ou para a versão do iOS. Isso significa que ele é impossível de ser estilizado com bordas arredondadas exatas, sombras de gradiente ou fontes exóticas. Para corporações reais, criamos botões customizados empacotando uma camada invisível de toque, chamada TouchableOpacity (ou modernas Pressables), envolvendo um container texturizado. Ao ser tocada, o TouchableOpacity responde ao dedo providenciando opacidade visual simulando "aperto" antes de disparar o script.

A junção primária desses quatro componentes (a *View* agrupando flexbox horizontal, o *Text* guiando os rótulos, o *TextInput* esperando a digitação do teclado virtual, e um *TouchableOpacity* invocando uma API de nuvem e enviando os dados validados) resume mais de 60% da engenharia de front-end nativo em qualquer parte do planeta.

Por fim, a principal dor de cabeça em telas mobile baseadas em inserção de dados são os On Screen Keyboards (Teclado em Tela). Ao clicar em um TextInput no rodapé da página, o teclado digitalizado empurra a tela para cima nativamente sobrepondo de forma "cega" a camada onde os botões estão desenhados pelo JavaScript. Se prever essa variação vertical não for elaborada usando a ferramenta do KeyboardAvoidingView acoplada ou `ScrollView`, o aplicativo parecerá que é de baixíssima procedência comercial, impossibilitando o próprio usuário de clicar em submeter.

🛠️ Passo a Passo (Mão na Massa)

Trabalhe Iterativamente

1. Ligue o emulador (A.V.D.).
2. Deixe o Metro rodando via npx react-native start ou abra via run-android.
3. Salve o arquivo e assista instantaneamente a alteração pintando a tela (Fast Refresh).

1º Passo: Limportação Essencial
Crie uma nova tela em uma pasta recém criada da raiz /src/screens/LoginScreen.tsx e importe os fragmentos fundamentais do React e do construtor nativo.

2º Passo: Componente Funcional e Estado
Utilize o hook base useState para destinar memória para o futuro e-mail e password.

3º Passo: Layout Principal (View)
Delinear a View base com fundo sólido corporativo (Senai-blue ou Branco gelo) preenchendo a tela via Flex 1 e alinhamento centrado.

4º Passo: Formulários (Inputs Individuais)
Conecte os inputs de texto aos hooks controláveis, definindo os eventos onChangeText alterando pontualmente o setEmail(text) e o setPassword(text).

5º Passo: Customização do Botão Touchable
Envolva seu bloco num TouchableOpacity para chamar um validador simulado (que gera o Alert.alert do sistema).

6º Passo: Protegendo contra Senha Exposta
Ative a diretiva booleana nativa secureTextEntry={true} sobre o input da senha, indicando pro sistema Android que as bolinhas blindadas devem obscurecer as letras durante rastreio de interface (imune a gravadores digitais e cópias da área de transferência).

💻 Exemplos Analisados (O Código da Tela)

Exemplo 1: Controlando o Estado de Input

O foco em uma Squad é que a subversão da "DOM" seja proibida. React exige "Formulários Controlados", onde state domina o input.


// Exemplo parcial e focado unicamente no Estado e Input
import React, { useState } from 'react';
import { View, TextInput, StyleSheet } from 'react-native';

const CampoSeguro = () => {
  // useState() retorna primeiro o VALOR, e em seguida um MÉTODO DE ALTERAÇÃO.
  const [senha, setSenha] = useState("");

  return (
    <View style={styles.container}>
      <TextInput 
        style={styles.input}
        placeholder="Digite a Senha Corporativa"
        value={senha} // O campo reflete a memória estrita
        onChangeText={setSenha} // Ação do teclado atualiza a memória estrita
        secureTextEntry={true} // Diz ao sistema para colocar * nativo
      />
    </View>
  )
};
const styles = StyleSheet.create({
  container: { padding: 20 },
  input: { borderBottomWidth: 1, borderColor: '#ccc', height: 40}
});

Explicação linha a linha: Ao usarmos `value={senha}` amarramos a visualização. Ao usarmos o `onChangeText={setSenha}`, instruímos o framework que para CADA tecla individual batida internamente pelo teclado Android, o método invocará a nova renderização inteira do bloco, validando se o asterisco precisa ser montado. `secureTextEntry` não é um hack web de trocar fonte para bolinhas — ele invoca a proteção binária da API de InputMethodManager do próprio Linux do celular Android.

Exemplo 2: O Código Completo da Tela de Login

Criação unificada da funcionalidade abordando alertas e custom button (Botão SENAI).


import React, { useState } from 'react';
import { 
  View, Text, TextInput, TouchableOpacity, 
  StyleSheet, Alert, KeyboardAvoidingView, Platform 
} from 'react-native';

const LoginScreen = () => {
  const [email, setEmail] = useState('');
  const [senha, setSenha] = useState('');

  // Lógica desvinculada para proteger o evento sujo na UI.
  const manipularAcesso = () => {
    // Validação primitiva sanitizada
    if (email.trim() === '' || senha.trim() === '') {
      // Alert chama biblioteca nativa e trava a UI
      Alert.alert('Erro Inesperado', 'Campos vazios bloqueiam a operação. Preencha!');
      return; 
    }
    
    if (senha.length < 6) {
      Alert.alert('Inseguro', 'Senha requer mínimo de 6 caracteres operacionais.');
      return;
    }
    
    // Simulação do sucesso de Login na Infraestrutura
    Alert.alert('Sucesso', 'Token gerado! O sistema abriu a API com `'+ email +'`.');
  };

  return (
    <KeyboardAvoidingView 
      style={styles.geral} 
      behavior={Platform.OS === 'ios' ? 'padding' : 'height'}
    >
      <View style={styles.caixaFrontal}>
        <Text style={styles.heading}>Área do Empregado</Text>

        <TextInput 
          style={styles.inputCorporativo}
          placeholder="E-mail (ex: pcp@senai.br)"
          keyboardType="email-address" // Troca teclado para formato E-MAIL (aparece @)
          autoCapitalize="none" // E-mail inicial maiúsculo irrita e afeta lógica
          value={email}
          onChangeText={setEmail}
        />

        <TextInput 
          style={styles.inputCorporativo}
          placeholder="Senha Segura"
          secureTextEntry // Atalho nativo para true
          value={senha}
          onChangeText={setSenha}
        />

        <TouchableOpacity style={styles.botaoPrincipal} onPress={manipularAcesso}>
          <Text style={styles.botaoTexto}>AUTENTICAR ACESSO</Text>
        </TouchableOpacity>
      </View>
    </KeyboardAvoidingView>
  );
};

const styles = StyleSheet.create({
  geral: {
    flex: 1,
    backgroundColor: '#f4f7fb',
    justifyContent: 'center',
    padding: 24,
  },
  caixaFrontal: {
    backgroundColor: '#ffffff',
    padding: 32,
    borderRadius: 8,
    shadowColor: '#164194',
    elevation: 3, 
  },
  heading: {
    fontSize: 24,
    fontWeight: 'bold',
    color: '#0f2c66',
    textAlign: 'center',
    marginBottom: 32,
  },
  inputCorporativo: {
    borderWidth: 1,
    borderColor: '#e2e8f0',
    backgroundColor: '#f8fafc',
    borderRadius: 6,
    paddingHorizontal: 16,
    paddingVertical: 12,
    fontSize: 16,
    marginBottom: 16,
    color: '#1e293b'
  },
  botaoPrincipal: {
    backgroundColor: '#164194', // A base visual que requeremos no laboratório!
    paddingVertical: 16,
    borderRadius: 6,
    alignItems: 'center',
    marginTop: 8,
  },
  botaoTexto: {
    color: '#ffffff',
    fontWeight: 'bold',
    letterSpacing: 1.2, // Espaçamento providencial lido por bibliotecas fontes
  }
});

export default LoginScreen;

Discussões Analíticas da Prática: Por quê e como justificamos essas classes?

Na prática completa acima, notem o uso de propriedades de teclado `keyboardType="email-address"`. Um teclado Android regular tentará adivinhar a palavra, não mostrará símbolos e causará frustração se não injetarmos esse atributo. O `autoCapitalize="none"` é essencial: celulares insistem em botar E-mails com letra maiúscula inicial por se tratar de início de frase, e os bancos de dados modernos distinguem caracteres de autenticação. A `KeyboardAvoidingView` evita o problema que citamos lá da barra subir infinitamente de modo que abocanhasse a tela do emulador inteirinha.

🌟 5 Boas Práticas Profissionais (Componentes Base)

  • Componentização Inteligente: Ao invés de ficar replicando propriedades de cor em 30 caixas `TextInput` pelo sistema, crie um componente secundário customizado (ex: `<CustomInput>`) que retorne de forma empacotada o padrão corporativo com seu design contido.
  • Blindagem do autoCapitalize: Sempre defina autoCapitalize="none" e autoCorrect={false} em campos como E-mail, Senhas e Usernames corporativos onde validação em tempo real e de cache é requerida. O corretor destrói o stringing do banco se tentar sugerir em campos vitais.
  • Frequência de Padding, Nunca Hard Coded Height: Em `View` e `TextInput`, defina alturas usando `paddingVertical` preenchendo as arestas. O uso de `height: 48` restrito fixa botões em telas pequenas (emulador Píxel menor) ou esconde acessibilidade de leitura grande no Android (usuários de idade com fator de fonte 2.0 quebram layout que usou tamanho chumbado fixo).
  • Uso Responsável da Memória no OnChange: O `onChangeText` não requer passagem de evento anônimo via seta dupla: `onChangeText={(e) => setSenha(e)}`. A sintaxe `onChangeText={setSenha}` automaticamente repassa e resolve a execução de processador de the Bridge economizando alocação de callback anônimo a cada letra apertada.
  • Teclados Sem Retorno: Em formulários, utilize as propriedades auxiliares `returnKeyType="next"` junto com `onSubmitEditing` no primeiro campo pra focar o segundo e gerar navegação rápida e sem cliques indesejados.

🚧 5 Erros Comuns e Frequentes do Ambiente Componentizado

1. Uso de ScrollView Incorreto esmagando Formulários

Usuários que colocam ScrollView aninhados esquecem o atributo `contentContainerStyle`. A ScrollView normal amassa um login pra ocupar zero pixels. A regra de flex: 1 precisa do content block específico para expandir.

2. Warning de "Encountered two children with the same key" numa View gerada

Ao se gerar mapeamentos usando Arrays em React e inserindo tags Text/View, o programador omite prop `key=""`. O Android enlouquece reajustando todas linhas a cada atualização de frame pois carece ID unívoco iterável.

3. Sombras ausentes fora de iOS

As clássicas props web `shadowRadius`/`shadowColor` NÃO FAZEM efeito gráfico profundo nenhum no nosso A.V.D (Android Emulator). Você é obrigado a usar `elevation: 6` (api proprietária Android preta Material Design) simultaneamente combinada para surgir a sombra no campo.

4. Botão esconde debaixo do teclado

Mesmo usando KeyboardAvoidingView a tela fica estática. Quase que 100% dos erros é atômico: o Behavior nativo requer "height" explícito em `Platform.OS === 'android'` versus o "padding" da versão web-based. A checagem condicional com `Platform` da API core resolve.

5. Uso de "div" por reflexo (Text inside View Sem Wrappers)

Diferente do React DOM pra web, você não pode escrever ` Olá Mundo `. Essa subversão desprovida do componente primitivo `Text` gera red screen fatal (exceção nativa violenta) relatando "Invariant Violation: Text strings must be rendered within a Text Component". Trata-se da violação do pacote Native Modules UI.

📍 Resumo Direto ao Ponto (Encerramento de Módulo)

  • Renderização Nativa: Tags como <View> e <Text> interagem diretamente com o hardware.
  • Manipulação Tátil: Uso de botões específicos e adaptativos nativos na plataforma (Touchables Interfaces).
  • Tratamento de Teclados: A obrigatoriedade absoluta do KeyboardAvoidingView para que a interface e as páginas não quebrem no mobile.
  • Gerenciamento de Estado: Controle central puro da aplicação em execução com a base do ecossistema reativo no uso do useState.
  • Sintaxe Segura: Evitar renderizações de funções injetadas e anônimas nas props para não acarretar penalidade à performance geral sobrecarregando de lixo a máquina local.
